# 龙虾飞书接入方案说明

## 1. 背景

当前讨论的“龙虾”指本地部署的 HexClaw/龙虾引擎。目标是回答三个问题：

1. 龙虾现在是如何和飞书连接的？
2. 为什么当前桌面版很难做到“安装后直接使用”？
3. 如果希望用户安装客户端后尽量开箱即用，推荐采用什么方案？

本文基于当前本机代码、运行配置和运行时状态整理。

## 2. 龙虾当前如何连接飞书

### 2.1 基本原理

龙虾接飞书的核心链路是：

1. 飞书收到用户消息
2. 飞书服务器把事件回调到龙虾的 Webhook
3. 龙虾解析飞书事件 `im.message.receive_v1`
4. 龙虾把消息转换成内部统一消息结构，交给引擎处理
5. 引擎生成回复
6. 龙虾再使用 `app_id + app_secret` 换取 `tenant_access_token`
7. 龙虾调用飞书 OpenAPI 将回复发回飞书

也就是说：

- 入站：`飞书 -> 龙虾 webhook`
- 出站：`龙虾 -> 飞书 OpenAPI`

### 2.2 代码位置

飞书适配器实现位于：

- `adapter/feishu/feishu.go`

其中包含这些能力：

- 处理 `challenge` 校验
- 处理 `im.message.receive_v1`
- 校验 `verification_token`
- 获取 `tenant_access_token`
- 调用 `/im/v1/messages` 发消息

### 2.3 两种挂载方式

龙虾后端支持两种飞书入口形式：

#### 方式 A：独立 Webhook 服务

飞书适配器可以直接监听：

- `POST /feishu/webhook`

默认注释里使用独立端口 `6061`。

#### 方式 B：实例化挂载到主 API

当前新版实例管理会把飞书实例挂到主服务：

- `GET /api/v1/platforms/hooks/{provider}/{name}`
- `POST /api/v1/platforms/hooks/{provider}/{name}`

对于飞书实例，实际入口类似：

- `/api/v1/platforms/hooks/feishu/飞书`

当前本机运行时采用的是这条方式。

## 3. 当前本机龙虾的实际形态

### 3.1 当前监听方式

本机运行配置显示：

- `server.host: 127.0.0.1`
- `server.port: 16060`

也就是当前龙虾只监听本机：

- `127.0.0.1:16060`

这意味着：

- 浏览器/桌面端本机可以访问
- 飞书云端默认不能直接访问

### 3.2 当前飞书实例状态

本机运行时数据库中已有一个飞书实例：

- provider: `feishu`
- name: `飞书`
- mode: `webhook`

桌面端保存的飞书配置包含：

- `app_id`
- `app_secret`
- `verification_token`

### 3.3 当前实际问题

当前链路虽然已经把飞书实例挂到了：

- `/api/v1/platforms/hooks/feishu/飞书`

但运行时健康检查依然报错：

- `feishu app_id/app_secret 未配置`

根因是桌面端同步给后端实例管理时，保存的是 `snake_case`：

```json
{
  "app_id": "...",
  "app_secret": "...",
  "verification_token": ""
}
```

而当前后端实例结构在运行时反序列化阶段实际依赖的是 `AppID/AppSecret` 这一层映射结果，导致“实例保存成功，但适配器读取为空”。

换句话说，当前桌面版存在一个运行时字段映射问题。

## 4. 为什么当前桌面版做不到“安装即用”

当前桌面版的本质架构是：

1. 客户端本地起 sidecar
2. sidecar 以 `--desktop` 模式运行
3. 只监听 `127.0.0.1:16060`
4. 飞书 webhook 理论上挂到本地 sidecar 的 API 上

问题在于，飞书收消息依赖飞书服务器主动回调你的 webhook。

如果客户端只监听 localhost，那么：

- 飞书服务器打不到用户本机
- 就算本地飞书实例配置正确，也无法收到飞书回调

因此当前模式下，用户如果想收飞书消息，额外还需要：

- 可公网访问的域名或隧道
- 回调地址配置
- 飞书应用配置
- 本地 sidecar 必须持续在线

这显然不属于“安装即用”。

## 5. 两种可行产品方向

## 5.1 方案一：客户端内置隧道

### 方案描述

保留“用户本地跑龙虾”的方式，但由客户端自动启动内网穿透，比如 `ngrok`。

链路变为：

1. 本地 sidecar 启动
2. 客户端自动启动隧道
3. 获取一个公网 HTTPS 地址
4. 飞书事件订阅指向该地址
5. 公网请求经隧道转发到本地 webhook

### 优点

- 改动相对较小
- 仍然保留本地执行、本地数据
- 适合小规模用户验证

### 缺点

- 每个用户都需要一个公网 tunnel
- 成本随在线用户数增长
- tunnel 掉线时 webhook 立即失效
- 域名和 token 管理复杂
- 不是长期最佳实践

### 适用场景

- 小规模内测
- 强调本地执行
- 追求最快落地

## 5.2 方案二：中心化接入层

### 方案描述

把飞书 webhook 统一接到你们的公网服务，而不是用户本机。

链路变为：

1. 飞书 webhook 打到你们公网 HexClaw 服务
2. 公网服务完成鉴权、消息接收、实例路由
3. 由服务端直接执行，或再通过长连接转给本地客户端执行
4. 服务端统一回复飞书

### 优点

- 最接近“安装即用”
- 用户无需自行配置 tunnel
- 安全、审计、运维集中
- callback URL 稳定
- 更适合正式商用

### 缺点

- 服务端改造量更大
- 需要公网部署、域名、证书、监控
- 若追求本地执行，还需要额外的客户端回连协议

### 适用场景

- 正式商用
- 希望普通用户安装后直接使用
- 希望统一管理飞书应用和接入策略

## 6. “安装后直接使用”应该怎么做

如果目标是让普通用户“安装软件就能直接使用”，推荐优先采用：

- 中心化接入层方案

最小可行版本建议是：

1. 飞书 webhook 统一接入公网服务
2. 飞书应用凭证由平台统一管理
3. 用户安装客户端后登录账号
4. 客户端只作为控制台和会话界面
5. 飞书消息先由服务端执行并回包

后续如果希望强化“本地隐私 / 本地工具 / 本地模型”，再升级为：

- 公网接入层 + 本地执行

即：

1. 飞书 webhook 仍然先打到公网服务
2. 公网服务通过长连接把消息转发到用户本地客户端
3. 本地客户端处理后把结果回传服务端
4. 服务端再回复飞书

这样既能安装即用，也能保留本地执行能力。

## 7. 对当前桌面版的直接建议

在当前代码基础上，短期和中期建议分开：

### 7.1 短期建议

修复当前桌面版的两个明显问题：

1. 修复飞书实例配置同步到运行时的字段映射问题
2. 在 IM 通道页面展示真实 webhook 路径，并明确提示：
   - 桌面版默认只监听 localhost
   - 如无公网入口，飞书无法回调到本机

### 7.2 中期建议

如果仍坚持本地执行：

1. 内置隧道能力
2. 自动生成公网 callback URL
3. 自动健康检查 tunnel 和 sidecar

### 7.3 长期建议

正式产品建议迁移到：

- 中心化公网接入层

这是最接近最佳实践、也最容易实现“安装即用”的方向。

## 8. 总结

龙虾本身已经具备飞书接入能力，核心逻辑没有问题：

- 能收 webhook
- 能解析飞书消息
- 能调用飞书 API 回复

当前真正的瓶颈不在“有没有飞书适配器”，而在：

1. 桌面端运行时字段映射存在 bug
2. 桌面版默认只监听 localhost
3. 没有公网入口就无法让飞书回调到用户本机

因此：

- 龙虾的飞书接入机制可以继续用
- 但如果目标是“用户安装后就能用”，最终还是要补齐公网接入层，或至少补齐自动隧道能力
