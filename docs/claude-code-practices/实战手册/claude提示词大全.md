## 一、验证驱动：官方排名第一的最佳实践

> 给 Claude 验证手段是最高杠杆率技巧。Claude 能自我验证时，输出质量会大幅提升。

### 提供测试作为验证

```
重构 auth 中间件，使用 JWT 替代 session token。
改完后运行现有测试套件，修复所有失败用例后再结束。
```

### 提供截图/预期输出作为验证

```
修改登录页的表单布局，改完后截图给我看效果，确认没有 UI 回归。
```

### 不接受"我认为能运行"

```
请实际运行验证，而不是假设它能工作。
```

### 让 Claude 自我审查

```
仔细审查这些改动，确认没有遗漏后再提交。
```

## 二、先探索，再规划，再编码

> 用 `Shift+Tab` 切换 Plan 模式。Plan 模式下 Claude 只读不改，适合复杂任务前的分析。

### 复杂任务先做规划

```
进入 plan 模式。分析支付模块的架构，给出重构方案，
列出影响范围和风险点，等我确认后再动手。
```

### 探索式分析

```
不要改任何代码。先通读 src/auth/ 目录，画出调用链路，
告诉我当前的问题在哪，以及你的修复思路。
```

### 面试式需求澄清

```
先不要写代码。先问我足够多的问题来理解需求，
直到你有信心后再开始实现。
```

## 三、划定范围，避免漫无边际

> 不要让 Claude 无限制探索，明确范围能显著提升准确性。

```
❌ 找出这个代码库中所有问题

✅ 在 src/auth/ 目录下，找出认证逻辑中最多 3 个安全漏洞
```

### 指向具体文件和模式

```
参考 src/adaptor/vendor_a/adaptor.go 的实现模式，
给 vendor_b 写一个同样结构的适配器。
```

### 描述症状而非猜测原因

```
❌ Redis 连接池可能有泄漏，修一下

✅ 压测 10 分钟后 Redis 连接数从 10 涨到 200 不回落，
   报错日志见 @error.log，请排查根因。
```

### 提供项目背景再下达指令

```
项目背景：Go + Gin API，K8s 多 Pod 部署
当前问题：渠道健康检查偶尔误判，导致不必要的熔断
约束：不改数据库 Schema

基于以上背景，优化健康检查的判定逻辑。
```

## 四、自定义命令：封装常用提示词

> 重复使用的复杂提示词封装成 `.claude/commands/` 下的命令文件，一个斜杠即可调用。

### 封装审查提示词

```markdown
<!-- ~/.claude/commands/review-security.md -->
对当前分支的全部变更进行安全审查：

1. SQL 注入、命令注入、XSS
2. 敏感信息泄漏（日志/响应/前端）
3. 认证鉴权绕过
4. 输入校验缺失

每个问题标注严重等级，给出修复建议。
$ARGUMENTS
```

使用：
```
/review-security
/review-security 只看 src/handler/ 目录
```

### 封装模板生成命令

```markdown
<!-- .claude/commands/new-adaptor.md -->
参考 src/adaptor/vendor_a/adaptor.go 的实现模式，
为 $ARGUMENTS 创建一个新的适配器。

必须包含：
1. adaptor.go — 实现 Adaptor 接口
2. constants.go — 常量和配置列表
3. 在 adaptor_factory.go 中注册
4. 在 constants/ 目录中添加类型常量
```

使用：
```
/new-adaptor vendor_b
```

> `$ARGUMENTS` 是唯一占位符，会被命令后面的参数替换。

## 五、图片/截图输入：多模态提示词

> Claude 是多模态模型，可以直接看懂截图、架构图、UI 设计稿。

### UI Bug 定位

```
看这个截图 @screenshot.png，登录按钮点击后没反应。
检查对应的 onClick handler 和 API 调用链路，找出问题。
```

### 用设计稿实现 UI

```
参考这张设计稿 @design.png，用 React + Tailwind 实现这个页面。
要求响应式布局，颜色和间距尽量还原。
```

### 看报错截图排查

```
这是生产环境的报错截图 @error.png，分析根因并给出修复方案。
```

> 支持 PNG/JPG/WebP/GIF，拖拽到对话框或用 `@` 引用均可。

## 六、子代理：并行调研不污染主上下文

> 大型调研任务委派给子代理，保持主会话干净高效。

### 并行调研

```
帮我并行调研以下 3 个问题：
1. 项目中所有定时任务的实现方式
2. auth 中间件的完整调用链
3. 最近 20 次提交涉及的模块分布
```

### 委派重型分析

```
派一个子代理去分析 src/adaptor/ 目录下所有适配器的错误处理模式，
列出不一致的地方，不要在主会话中展开。
```

## 七、反馈循环：让 Claude 自我修正

### 内置验证机制

```
重构 auth 中间件，使用 JWT 替代 session token。
改完后运行现有测试套件，修复所有失败用例后再结束。
```

### 对每个决策提出质疑（Code Review 模式）

```
找出 main 和当前分支的差异，对每一个改动提出质疑：
- 为什么选这个方案而不是替代方案？
- 有没有边界情况没覆盖？
- 性能影响如何？
```

### 管道输入做 Review

```bash
git diff main | claude "review 这个 diff，指出潜在问题"
```

## 八、推倒重来，追求优雅

当方案走入死胡同时，丢弃已有包袱，重新用最优解实现：

```
鉴于你现在所掌握的一切信息，放弃这个方案，实施更优雅的解决方案。
```

> **适用场景：** 调试陷入僵局、方案越补越复杂、技术债务累积时。
> **搭配使用：** `/rewind` 回滚到改动前的状态，避免残留脏代码。

## 九、结构化提示词（复杂任务）

> 当单条指令不足以表达复杂需求时，用结构化格式拆分。

### Markdown 格式（CLI 推荐）

```
## 任务
分析认证代码中的安全漏洞

## 重点关注
- 密码哈希处理
- SQL 注入风险

## 约束
- 不修改现有数据库 Schema
- 兼容 Go 1.25+

## 输出要求
- 每个漏洞标注严重等级
- 给出修复后的代码
- 按业务影响排优先级
```

### XML 格式（API 调用/自动化脚本推荐）

```xml
<task>分析认证代码中的安全漏洞</task>
<focus>
  - 密码哈希处理
  - SQL 注入风险
</focus>
<constraints>
  - 不修改现有数据库 Schema
</constraints>
<output>
  - 每个漏洞的严重等级
  - 修复后的代码示例
</output>
```

> XML 格式更适合 API 调用场景；CLI 交互中 Markdown 更自然。

## 十、让 Claude 学习陌生工具

遇到 Claude 不熟悉的 CLI 工具，让它自学：

```
使用 'foo-cli --help' 了解该工具的用法，然后用它完成以下任务：[具体描述]
```

也可以指向文档：

```
阅读 @docs/api-spec.yaml，基于这个规范生成 Go client 代码。
```

## 十一、反模式（避免踩坑）

```
❌ 一个提示词塞太多不相关任务
   → 应拆分任务，每个任务前 /clear

❌ "改善代码质量" / "找出所有问题"
   → 太模糊，给具体目标和范围

❌ 在提示词中引用内部工具名（如 AskUserQuestion、Agent）
   → 用自然语言表达意图，Claude 会自己选择工具

❌ 不给验证手段就说"确保没有 bug"
   → 提供测试命令、预期输出、截图对比

❌ 一次让 Claude 改 50 个文件
   → 分批修改，每批验证通过后再继续

❌ 所有规则都写在提示词里反复说
   → 确定性行为用 Hooks，持久规则写 CLAUDE.md
```

## 附录：提示词模板库

### A. Bug 修复模板

```
## Bug 描述
用户登录后 token 过期时间不正确，应为 24h 实际只有 1h。

## 复现步骤
1. 登录获取 token
2. 解码 JWT 查看 exp 字段

## 预期结果
exp = iat + 86400

## 相关文件
@middleware/auth.go
@config/jwt.go

## 验证方法
运行 go test ./middleware/... -run TestTokenExpiry
```

### B. 新功能模板

```
## 需求
为渠道管理添加健康检查自动熔断功能。

## 参考实现
参考 @src/adaptor/vendor_a/adaptor.go 的模式

## 约束
- 多 Pod 安全（不依赖本地状态）
- 熔断阈值通过环境变量配置
- 需要单元测试

## 验收标准
1. 连续 3 次失败触发熔断
2. 熔断后每 60s 尝试恢复
3. 所有测试通过
```

### C. 代码审查模板

```
审查 @src/router/request_router.go 最近的改动：

1. 正确性：逻辑有没有漏洞？边界情况覆盖了吗？
2. 安全性：有注入风险吗？敏感信息处理得当吗？
3. 性能：有 N+1 查询或不必要的内存分配吗？
4. 可维护性：命名清晰吗？逻辑易读吗？

只指出问题，不要改动代码。
```

### D. 迁移/重构模板

```
## 目标
将 channel 配置从硬编码迁移到数据库驱动

## 当前状态
@src/constants/channel.go 中有 30+ 常量，每次新增需要改代码发版

## 约束
- 必须向后兼容现有 channel ID
- 迁移期间不能中断服务
- 支持回滚

## 步骤要求
1. 先出迁移方案（不要直接改代码）
2. 列出所有受影响文件和调用链
3. 给出 SQL 迁移脚本
4. 给出灰度/回滚方案
```

### E. 性能排查模板

```
## 症状
API /api/resource/list 响应从 50ms 劣化到 800ms，
发生在最近 3 天，coincides with 数据量从 50 增加到 200。

## 已知信息
- 数据库：SQLite，单文件
- 无缓存层
- 日志见 @perf.log

## 排查要求
1. 用 pprof 或日志定位瓶颈（CPU/IO/DB）
2. 检查是否存在 N+1 查询
3. 检查是否缺索引
4. 给出优化方案和预期提升
```

### F. 多模态输入模板

```
## UI 还原
参考设计稿 @design.png，用 React + Tailwind 实现。

## 要求
- 颜色、间距、字号尽量还原
- 响应式适配（桌面 + 移动端）
- 组件拆分合理，可复用

## 验证
实现后截图给我对比设计稿差异。
```
