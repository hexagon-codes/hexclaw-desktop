# 设计驱动 · 测试闭环 · 多 Agent 协作

> Claude 做方案设计和编码实现，Codex 做 Code Review 和文档评审，测试验证闭环贯穿始终。
> 纯提示词/工作流参考，功能操作请看《claude使用手册.md》。

## 核心理念

### 三大支柱

```
设计驱动          测试闭环              多 Agent 协作
   │                 │                      │
   ▼                 ▼                      ▼
PRD → 需求拆解    每个阶段都有          Claude 设计+编码
 → API 设计       可验证的测试            Codex 审查+补测试
 → 多方案对比     覆盖正常/异常/边界      交叉评审消除盲区
 → ADR 决策       测试不过不算完成        对照设计文档验闭环
```

### 分工模型

| 角色 | 负责 | 工具 |
|------|------|------|
| **Claude** | 方案设计、架构分析、编码实现、TDD | Claude Code CLI |
| **Codex** | Code Review、测试盲区补充、设计文档评审 | Codex CLI |
| **人类** | PRD 输入、方案决策、最终确认 | 你 |

### 完整流程

```
1. 人类输入 PRD
2. Claude 解读需求 → 拆解 → 技术转化 → API 设计
3. Claude 出 2-3 个架构方案 → 人类选定 → ADR 记录
4. Claude TDD 实现（红→绿→重构）
5. Codex Code Review（安全/性能/可维护性）
6. Claude 回应审查意见（接受/反驳/讨论）
7. Codex 补测试盲区 → Claude 修复发现的 bug
8. Claude + Codex 对照设计文档做功能闭环评审
9. 人类最终确认 → 上线
```

### 三条铁律

1. **先设计后编码**：不直接写代码。先读 PRD → 拆需求 → 出方案对比 → ADR 决策 → 再实现。设计阶段投入的时间，在实现阶段 10 倍回报。
2. **写代码的和审代码的不是同一个"大脑"**：Claude 编码，Codex 审查。不同模型的训练数据和推理偏好不同，交叉审查能发现单一模型的盲区。
3. **没有测试验证不算完成**：每个阶段都要有可运行的验证。禁止"should pass"、"probably OK"——跑了测试/拿到证据才算。

### 适用场景

| 场景 | 流程 |
|------|------|
| 新功能开发 | PRD → 设计 → Claude 实现 → Codex 审查 → 闭环评审 → 上线 |
| 重构改造 | 分析 → 方案 → Claude 有保护网地重构 → Codex 审查 |
| Bug 修复 | Claude 定位 → 写复现测试 → 修复 → Codex 确认 → 回归 |
| 架构评审 | Claude 分析 + Codex 独立评审 → 交叉对比 → 人类决策 |

## 一、从 PRD 到架构：需求驱动设计

> 架构不是凭空设计的，必须从产品需求出发。先读懂 PRD，再整理需求，再推导架构。

### PRD 解读与需求提取

```
阅读以下产品需求文档（PRD），提取技术实现需要关注的信息：

@docs/prd-order-timeout.md

请输出：
1. 核心用户故事（按优先级排序）
2. 功能需求清单（每条标注：必须/应该/可以）
3. 非功能需求（性能/安全/可用性/兼容性）
4. 隐含约束（PRD 没明说但技术上必须考虑的）
5. 模糊点清单（需要和产品确认的问题）

不要开始设计，先把需求理清楚。
```

### 需求拆解与优先级

```
基于提取的需求，做进一步拆解：

1. 按 MoSCoW 分级：
   - Must have（没有就不能上线）
   - Should have（应该有，但可以二期）
   - Could have（锦上添花）
   - Won't have（本期明确不做）

2. 识别依赖关系：哪些需求必须先做？哪些可以并行？

3. 画出需求依赖图：
   A → B → D
   A → C → D
   （A 和 B/C 可以并行，D 依赖 B+C）

4. 给出建议的迭代计划（每个迭代交付哪些需求）
```

### 技术需求转化

```
把以下产品需求转化为技术需求：

产品需求：
- 用户下单后 30 分钟未支付自动取消
- 取消后库存自动回滚
- 用户能在"我的订单"看到取消原因

请输出：
1. API 设计（需要哪些接口、字段、状态码）
2. 数据模型（需要哪些表/字段/索引）
3. 后台任务（定时/异步/事件驱动）
4. 状态机（订单状态流转图）
5. 异常场景（超时/并发/重复/部分失败）
6. 监控指标（需要观测什么）

每项标注来源于 PRD 的哪条需求。
```

### 从需求推导架构方案

```
基于以上技术需求，推导架构方案：

## 输入
- 技术需求清单（上一步输出）
- 现有代码库架构（先用 Plan 模式分析）
- 技术约束：Go + Gin / K8s 多 Pod / SQLite（按实际项目替换）

## 输出要求
1. 在现有架构上如何扩展（优先改造而非新建）
2. 需要新增哪些模块/接口/表
3. 需要修改哪些现有模块
4. 给出 2 个方案的对比（复杂度 vs 扩展性）
5. 推荐方案的实现步骤（按需求优先级排序）
6. 风险评估（哪些地方最可能出问题）

约束：方案必须能回溯到 PRD 需求——每个设计决策说明它满足哪条需求。
```

### 需求变更影响评估

```
产品提了一个需求变更：
[描述变更内容]

请评估影响：

1. 影响面分析：
   - 哪些已有模块需要修改？
   - 哪些接口契约会变？
   - 哪些测试需要更新？
   - 前端是否需要同步改动？

2. 工作量评估（AI 开发速度）：
   - 后端改动量
   - 前端改动量
   - 测试补充量

3. 风险评估：
   - 是否引入回归风险？
   - 是否影响已上线功能？
   - 是否需要数据迁移？

4. 建议：
   - 接受 / 推迟到下一版 / 需要重新设计
   - 如果接受，最小改动方案是什么？
```

### PRD → 架构完整流程

```
完整流程（按顺序执行）：

1. 阅读 PRD → 提取需求（不设计）
2. 需求拆解 → MoSCoW 分级 + 依赖关系
3. 产品确认 → 解决模糊点（输出给产品的问题清单）
4. 技术需求转化 → API/数据模型/状态机/异常场景
5. 分析现有架构 → Plan 模式读代码（不改）
6. 推导架构方案 → 2-3 个方案对比
7. 方案评审 → 可选 Claude + Codex 对抗评审
8. ADR 记录 → 记录选定方案和理由
9. 渐进式拆分 → 按需求优先级拆迭代
10. TDD 实现 → 每个迭代：红-绿-重构
```

## 二、架构分析：先读懂再动手

> 在 Plan 模式下（Shift+Tab）让 Claude 只读不改，先建立全景认知。

### 模块调用链分析

```
不要改任何代码。分析 src/service/ 目录的完整架构：
1. 画出模块间的依赖关系图
2. 列出每个 adaptor 实现的接口方法
3. 标注数据流向（请求→处理→响应）
4. 找出耦合最严重的 3 个地方
```

### 入口追踪

```
从 router/api.go 开始，追踪 POST /api/v1/orders 的完整链路：
请求 → 中间件 → handler → service → repository → DB → 响应

每一层列出：函数名、文件路径、关键参数、错误处理方式。
覆盖成功路径和失败路径。
```

### 陌生代码库快速上手

```
我刚接手这个项目。请分析：
1. 核心业务流程（从用户请求到最终响应）
2. 关键抽象（接口、适配器、工厂）
3. 配置管理方式
4. 测试覆盖情况
5. 最可能踩坑的地方

输出一份 500 字以内的架构速览。
```

### 依赖层级检查

```
检查当前项目的 import 依赖是否符合分层架构：
- controller 不应该直接 import model 的内部方法
- service 不应该直接 import controller
- 工具包不应该 import 业务包

列出所有违反分层的 import。
```

### 技术债评估

```
评估当前代码库的技术债：

1. 扫描所有 TODO/FIXME/HACK 注释，按模块统计
2. 找出超过 500 行的文件，评估是否需要拆分
3. 找出被 3 个以上模块依赖的"上帝包"
4. 找出没有测试的核心路径
5. 按修复优先级排序（影响面 × 修复难度）

输出技术债清单，附带建议的还债顺序。
```

## 三、API 接口设计

> PRD 转成技术需求后，先把 API 契约定下来——它是前后端、服务间协作的"合同"。

### RESTful 接口设计

```
基于以下技术需求，设计 RESTful API：

技术需求：
- 用户可以创建、查询、取消订单
- 支持按状态筛选和分页
- 管理员可以查看所有用户的订单

请输出：
1. 资源命名（名词复数，层级不超过 3 层）
2. HTTP 方法映射（CRUD → GET/POST/PUT/PATCH/DELETE）
3. URL 路径设计
4. 每个接口的请求参数和响应结构（JSON Schema）
5. 分页方案（游标 vs 偏移量，选哪个？为什么？）
6. 筛选/排序的查询参数设计
7. 鉴权方式（哪些接口需要登录？哪些需要管理员权限？）

约束：
- 保持与现有 API 风格一致
- 考虑未来的版本兼容
```

### 错误码体系设计

```
为项目设计统一的错误码体系：

要求：
1. HTTP 状态码使用规范（200/201/400/401/403/404/409/422/429/500）
2. 业务错误码格式（如 ORDER_001, USER_002）
3. 错误响应统一结构：
   { "code": "ORDER_001", "message": "用户可读的提示", "detail": "开发者排查信息" }
4. 错误码分类（参数校验/鉴权/业务冲突/系统错误）
5. 国际化考虑（message 是否支持多语言）

给出错误码文档模板和 Go 的实现代码。
```

### API 版本策略

```
项目需要支持 API 版本管理。对比以下方案：

1. URL 路径版本：/api/v1/orders → /api/v2/orders
2. Header 版本：Accept: application/vnd.api.v2+json
3. 查询参数版本：/api/orders?version=2

对比维度：
- 客户端升级成本
- 路由实现复杂度
- 缓存友好性
- 文档可读性

推荐方案 + 版本迭代策略（何时 v2？废弃 v1 的流程）
```

### 接口文档生成

```
基于路由定义和 handler 代码，自动生成 API 文档：

1. 扫描所有路由注册（router/*.go）
2. 提取每个接口的：路径、方法、请求参数、响应结构、鉴权要求
3. 输出 OpenAPI 3.0 格式的 YAML
4. 标注哪些接口缺少参数校验或文档注释

输出到 docs/openapi.yaml
```

## 四、方案设计：多方案对比决策

> 不要让 Claude 直接给"最佳方案"，让它给 2-3 个方案并对比。

### 方案对比模板

```
需求：为服务调用添加自动熔断功能。

请给出 2-3 个实现方案，每个方案包含：
1. 架构设计（组件图）
2. 核心数据结构
3. 关键流程
4. 优点和缺点
5. 多 Pod 安全性评估
6. 实现复杂度（1-5）

最后用表格对比，但不要替我做决定。
```

### 技术选型对比

```
需要为异步任务选择方案，对比以下选项：
1. Asynq (Redis-based)
2. 数据库轮询
3. Go channel + goroutine

对比维度：
- 多 Pod 安全性
- 故障恢复能力
- 运维复杂度
- 现有代码改动量
- 监控可观测性
```

### 用 ADR 记录架构决策

```
为刚才的方案选择写一份 ADR（Architecture Decision Record）：

## 标题
## 状态：已决定
## 背景：为什么需要做这个决策
## 考虑的方案：（列出所有方案）
## 决定：选择了哪个方案
## 理由：为什么选它
## 后果：这个决定带来的影响
```

### 渐进式设计

```
这个功能太大了，不能一步到位。请设计渐进式实现路径：

最终目标：完整的服务健康监控系统

请拆分成 3-4 个阶段，每个阶段：
1. 交付什么（最小可用功能）
2. 依赖什么（前置条件）
3. 验收标准
4. 可以独立上线吗？

约束：每个阶段不超过 3 天工作量（AI 开发速度）。
```

## 五、架构审查：发现劣化趋势

### SOLID 原则检查

```
审查 src/factory/adaptor_factory.go，从 SOLID 角度评估：

S - 是否单一职责？一个函数干了几件事？
O - 新增实现是靠加 case 还是靠扩展？
L - 子类型替换是否安全？
I - 接口是否最小化？有没有强迫实现者实现用不到的方法？
D - 高层是否依赖抽象？

每个违反点给出具体代码行号和改进建议。
```

### 反模式扫描

```
扫描 src/handler/ 目录，找出以下反模式：
1. God Object：单个文件/结构体超过 500 行
2. 复制粘贴扩散：高度相似的代码块
3. 大 switch/if-else 链：超过 10 个分支
4. 过度抽象：只有一个实现的接口
5. Feature Envy：频繁访问其他包的内部细节

每个问题标注严重程度和重构方向。
```

### 架构边界检查

```
检查当前架构是否存在边界侵蚀：
1. controller 层有没有在做业务逻辑（应在 service）
```

## 六、TDD 工作流：红-绿-重构

> 核心原则：先写失败的测试 → 写最少代码让测试通过 → 重构。

### 标准 TDD 提示词

```
用 TDD 方式实现服务健康检查的熔断逻辑：

第一步（红）：先写测试，覆盖以下场景：
- 连续 3 次失败触发熔断
- 熔断期间所有请求直接拒绝
- 熔断 60s 后进入半开状态
- 半开状态成功 1 次则恢复
- 半开状态失败则重新熔断
- 并发安全

第二步（绿）：写最少代码让所有测试通过
第三步（重构）：优化代码结构，确保测试仍然通过

每一步都要实际运行测试并展示结果。
```

### 为已有代码补测试

```
@src/service/health_check.go 这个文件没有测试。

1. 先分析所有公开方法的行为
2. 为每个方法写表驱动测试，覆盖：
   - 正常输入
   - 边界值（空值/零值/最大值）
   - 错误输入
   - 并发场景
3. 运行 go test -race 确认无竞态
4. 输出覆盖率报告
```

### Bug 修复的 TDD 方式

```
Bug：熔断恢复后计数器没有重置，导致恢复后一次失败就再次熔断。

请用 TDD 方式修复：
1. 先写一个能复现此 Bug 的测试（应该 FAIL）
2. 运行测试确认确实 FAIL
3. 修复代码
4. 运行测试确认 PASS
5. 运行全量测试确认无回归
```

### 表驱动测试模板

```
为 CalculateScore() 写表驱动测试：

要求：
- 用 Go 标准的 table-driven test 格式
- 每个 case 有清晰的命名（name 字段）
- 包含 setup/teardown 如果需要
- 测试数据覆盖正常/边界/异常
- 用 t.Parallel() 确认并发安全
```

### 契约测试

```
src/handler/order_handler.go 和前端有 API 契约。

1. 从路由定义中提取所有 API 的请求/响应结构
2. 为每个 API 写契约测试：
   - 请求参数校验（必填/类型/范围）
   - 响应字段完整性
   - 状态码正确性（200/400/401/404/500）
   - 错误响应格式一致性
3. 确保前端依赖的字段名和类型不被意外修改
```

### TDD 的适用边界

TDD 不是银弹，以下场景更适合"先实现再补测试"：

```
适合 TDD 的场景：
✅ 有明确输入输出的纯逻辑（计算/转换/校验/状态机）
✅ Bug 修复（先写复现测试 → 修复 → 验证）
✅ 接口契约（先定义请求响应结构 → 再实现 handler）
✅ 算法/数据结构

不适合 TDD 的场景：
⚠️ 探索性开发（需求不明确，边做边想）→ 先 spike 再补测试
⚠️ UI/前端交互（行为难用单元测试描述）→ 用 E2E 测试替代
⚠️ 第三方集成（依赖外部 API 行为）→ 先集成跑通再抽 mock
⚠️ 一次性脚本/数据迁移 → 验证结果即可，不需要持久测试

务实原则：TDD 是工具不是信仰。核心逻辑用 TDD，胶水代码补测试。
```

## 十三、多 Agent 协作：Claude + Codex 交叉审查

#### 模式一：写-审分离（日常开发）

```
3. Claude TDD 实现
4. Codex 补充测试盲区
5. Codex 做 code review
6. Claude 回应审查意见并修复
7. 人类最终确认 → 合并
```

> **核心原则**：写代码的和审代码的不应该是同一个"大脑"。
> 即使两个都是 AI，不同模型的训练数据和推理偏好不同，交叉审查能发现单一模型的盲区。

### 设计文档驱动的功能闭环评审

> 实现完成后，让 Claude 和 Codex 各自对照设计文档检查代码是否真正闭环——不是"代码能跑"，而是"PRD 的每条需求都有代码落地"。

#### Claude 对照设计文档评审

```
阅读以下两份文档：
1. 设计文档：@docs/design-order-timeout.md
2. 当前代码变更：git diff main

请对照设计文档逐条检查代码实现：

## 检查维度

### A. 需求覆盖率
对设计文档中每条功能需求，标注：
- ✅ 已实现（给出对应代码位置）
- ❌ 未实现（遗漏）
- ⚠️ 部分实现（说明缺什么）

### B. 流程闭环
对设计文档中的每条业务流程，追踪完整链路：
- 入口 → 处理 → 存储 → 响应 → 前端展示
- 是否存在"代码存在但流程断裂"的情况？
  - 数据写了但没有读的地方
  - 状态变了但没有通知的地方
  - 接口有了但前端没调的地方

### C. 异常场景覆盖
设计文档中列出的异常场景，代码是否都有处理？
- 超时 / 并发 / 重复 / 部分失败
- 错误响应是否符合设计文档的错误码规范

### D. 非功能需求
- 性能要求是否满足？
- 安全约束是否落地？
- 监控指标是否埋点？

输出格式：需求覆盖率表格 + 未闭环项清单 + 修复建议
```

#### Codex 独立做同样的闭环检查

```bash
# 把设计文档和 diff 一起喂给 Codex
codex "阅读设计文档和代码变更，检查功能是否闭环：

设计文档：$(cat docs/design-order-timeout.md)
代码变更：$(git diff main)

逐条检查：
1. 每条需求是否有对应代码？
2. 每条流程是否走得通（入口→处理→存储→响应）？
3. 异常场景是否都有处理？
4. 有没有写了代码但设计文档里没提到的（可能是过度实现）？

输出：需求覆盖率表格 + 问题清单"
```

#### 对比两份评审结果

```

请对比两份结果：

3. 两者结论矛盾的地方 → 列出争议点，由我决定

输出最终的闭环状态报告。
```

## 十四、实战组合拳

### 组合一：新功能完整开发流

```
1. 阅读 PRD → 提取需求    # 需求拆解 + MoSCoW 分级
2. 技术需求转化            # API/数据模型/状态机/异常场景
3. Shift+Tab → Plan       # 分析现有架构
4. 设计方案对比            # 多方案评估（可选：Claude + Codex 对抗设计）
5. ADR 记录决策            # 记录选定方案和理由
6. 渐进式拆分阶段          # 拆成可独立上线的小阶段
7. TDD 实现               # 每个阶段：红-绿-重构
8. Codex 补测试盲区        # 交叉测试互补
9. 功能闭环评审            # Claude + Codex 对照设计文档检查闭环
10. /review-go            # Go 架构级审查
11. /test-dev             # 测试环境回归
```

### 组合二：Bug 修复流

```
1. 描述症状（不猜原因）
2. Claude 定位根因 → 画出调用链 → 覆盖所有分支
3. TDD 修复：先写复现测试（FAIL）→ 修复 → PASS
4. 变更辐射区扫描（同文件/同包是否有同类问题）
5. 回归测试（go test -race ./...）
6. 总结教训追加到 memory/debugging.md
```

### 组合三：重构流

```
1. Plan 模式分析现有架构 → 画调用链
2. 确认测试覆盖率 baseline（go test -cover）
3. 补充缺失测试（确保重构有保护网）
4. 设计目标架构 → ADR 记录
5. 分步重构（每步都跑测试，确认全绿）
6. 对比重构前后：代码行数/复杂度/覆盖率
```

### 组合四：架构评审流

```
1. 入口追踪：画完整调用链（成功+失败+降级）
2. SOLID 检查：逐条评估
3. 反模式扫描：找出坏味道
4. 复杂度评估：圈复杂度/认知复杂度
5. 多 Pod 安全评估：并发/幂等/状态
6. 技术债盘点：TODO/FIXME 统计
7. 可选：Codex 独立评审，交叉对比结论
8. 输出 ADR 记录决策
```

### 组合五：上线前检查流

```
1. /review-fullstack        # 全栈审查
2. Codex 交叉 review        # 第二双眼睛
3. go test -race -count=3   # 并发安全验证
4. go build ./...            # 编译验证
5. 多 Pod 安全评估           # 分布式安全检查
6. 零停机迁移方案（如有 DB 变更）
7. 回滚方案确认
8. /test-dev                # 测试环境回归
9. 上线观察检查项清单
```

### 组合六：紧急 Hotfix 流

```
1. 确认症状 + 影响面（几个用户？核心链路还是边缘？）
2. Claude 最小化修复（只改必须改的，不顺手重构）
3. Claude 写复现测试 → 验证修复有效
4. Codex 快速 review（只看修复点，不做全面审查）
5. go test -race ./...      # 回归验证
6. 上线 + 盯 5 分钟指标
7. 事后：补完整测试 + 根因分析 + 教训记录
```

> **Hotfix 铁律**：紧急不等于跳过测试。最小化修复 + 复现测试 = 最快且最安全的方式。"先上再说回头补测试"是技术债的头号来源。
