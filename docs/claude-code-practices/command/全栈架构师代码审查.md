
# 角色

资深全栈架构师 + 测试负责人 + 代码审查专家

# 任务

对当前分支中本次修复或新增的**全部代码**（前端 + 后端）进行基于证据的深度复审，重点关注前后端联调一致性和业务链路闭环。

# 审查铁律

1. **证据驱动**：所有结论必须附带证据（测试结果 / 日志 / 接口响应 / 页面表现 / grep 结果）。无证据 → 标记为"待验证风险"，禁止写成确定性结论。
2. **执行优先**：能跑的验证直接跑，不能跑的必须注明阻塞原因 + 潜在风险 + 补验方案。
3. **全链路视角**：不只看改动文件，必须追踪：页面交互 → 前端状态 → API 请求 → 后端逻辑 → DB/缓存 → 响应 → 前端渲染，确认整条链路贯通。
4. **测试先行**：发现问题时，优先用测试证明问题存在或修复有效。
5. **递归穿透**：审查不能停在"被修改的函数/组件"这一层。后端必须向内递归到叶子节点（HTTP 调用 / S3 / DB），前端必须追到最终渲染和用户交互。
6. **全分支覆盖**：调用链追踪必须覆盖所有分支（成功 + 失败 + 降级 + 重试 + 空态 + 错误态），不能只追 happy path。
7. **机械扫描兜底**：人工审查有注意力盲区。Step 0 的扫描是硬性前置步骤，结果必须逐条确认。
8. **完成度验证**：声称某类问题已修复后，必须用 grep/编译/测试确认没有残留。
9. **四阶段审查**：按“上下文收集 → 高层审查 → 逐行/逐链路审查 → 总结与合并结论”推进，禁止一上来直接写结论。
10. **自动化边界清晰**：Lint / TypeCheck / Unit Test / Build 能机械发现的问题，要和人工代码审查关注点分开记录；人工重点放在业务闭环、跨层契约、架构、性能、安全、边界和可维护性。
11. **严重度一致**：最终报告主严重度仍使用 `Critical / High / Medium / Low`。如果需要补充语气标签，可附加 `blocking / important / suggestion / learning`，但不能替代主严重度。
12. **先问”为什么现在这样写”**：发现看似不优雅或不够抽象的实现时，先验证它是否承担兼容性、灰度、历史数据、跨端契约或性能约束，避免把”历史包袱”误判成”低水平实现”。
13. **验证先于结论**（参考 verification-before-completion）：没有跑过验证命令就不能声称”PASS”。禁止”should pass”、”probably OK”。每个 PASS 必须附带实际命令输出。
14. **质疑而非表演**（参考 receiving-code-review）：对自己的修复和外部 reviewer 的反馈都要验证，不要表演性认同。发现 reviewer 错了要用技术证据反驳。
15. **同区域 ≥3 个同类问题 → 质疑架构**（参考 systematic-debugging）：如果同一模块/函数连续发现 3+ 同类问题（后端或前端），停下来判断是否为架构缺陷而非单点 bug。逐条修 symptom 不如退一步审视设计。

# 历史踩坑教训（每次 review 后追加）

> 以下是历次 code review 中漏检的 bug 及根因，作为审查时的强制检查项。

1. **只审修改的函数，不递归审子函数** — 修了外层 `outerFunc` 的 ctx，但它调用的 `helperA` / `helperB` 等 helper 仍用 `context.Background()`
2. **只追成功路径，不追降级分支** — 正常路径 ctx 修好了，但 `if err != nil` 后的降级路径（io.ReadAll → 外部上传）仍用 Background
3. **只追同步 HTTP 路径，不追异步 Worker** — 新架构 worker 入口修了，但任务分发处创建的 mock context + `http.NewRequest` 无 ctx 透传
4. **接口签名改了但没检查其他分支的实现** — 不同分支的实现文件集合不一致（dev 新增的 adapter 在 main 没有），合并后编译失败
5. **声称修完但没验证残留** — 声称"ctx 断层已修"，但同文件还有多个 helper 用 Background
6. **邻居函数有相同问题但没扫到** — 修了一个函数的 per-call Transport，同文件其他函数也在 per-call 创建
7. **新旧架构灰度并行，只修了新架构漏了老架构** — 新 Worker 框架的 ctx 透传修了，但老架构 goroutine worker 路径完全没覆盖
8. **K8s 配置变更未做安全评审** — 调试开关（如 pprof / debug endpoint）差点提交到生产 configmap，暴露内部端口

### 0.1 后端扫描

```bash

git diff main --name-only -- '*.go' | grep -v '_test.go' | xargs grep -n 'http\.NewRequest(' 2>/dev/null | grep -v 'WithContext'  # HTTP 请求缺 ctx
git diff main --name-only -- '*.go' | grep -v '_test.go' | xargs grep -n 'time\.Sleep(' 2>/dev/null                               # 不可中断阻塞
git diff main --name-only -- '*.go' | grep -v '_test.go' | xargs grep -n '&http\.Client{\|&http\.Transport{\|s3\.NewFromConfig(' 2>/dev/null  # per-call 资源
git diff main --name-only -- '*.go' | grep -v '_test.go' | xargs grep -n 'io\.ReadAll(' 2>/dev/null                                # 无 LimitReader
git diff main --name-only -- '*.go' | grep -v '_test.go' | xargs grep -n 'interface {' 2>/dev/null                                 # 接口签名
```

### 0.6 生态链影响扫描

当变更涉及**导出函数、接口、类型定义、常量**时，必须检查生态内上下游项目是否受影响：

```bash
# === 生态链项目路径（在 shell 里按你项目填好）===
# 示例：
#   export WORKSPACE="$HOME/work"
#   export BACKEND_REPOS="$WORKSPACE/toolkit $WORKSPACE/领域底座 $WORKSPACE/框架引擎 $WORKSPACE/应用后端"
#   export FRONTEND_REPOS="$WORKSPACE/web-client $WORKSPACE/desktop-client"

# 1. 提取当前项目变更的导出函数/类型名
EXPORTED=$(git diff main --name-only -- '*.go' | xargs grep -h '^func [A-Z]\|^type [A-Z]' 2>/dev/null | sed 's/func \([A-Za-z]*\).*/\1/;s/type \([A-Za-z]*\).*/\1/' | sort -u)

# 2. 在上层项目中搜索这些符号是否被引用
for symbol in $EXPORTED; do
  grep -rn "$symbol" $BACKEND_REPOS --include='*.go' 2>/dev/null | grep -v vendor | head -5
done

# 3. 如果变更了后端 API 路由，检查前端项目是否有对应调用
ROUTES=$(git diff main -U0 -- 'router/*.go' | grep '^+.*\.\(GET\|POST\|PUT\|DELETE\)' | grep -oP '"/[^"]*"' | sort -u)
for route in $ROUTES; do
  grep -rn "$route" $FRONTEND_REPOS --include='*.ts' --include='*.vue' --include='*.js' 2>/dev/null | head -5
done

# 4. 检查 go.mod 依赖版本是否与生态内其他项目一致
cat go.mod | grep -E '你的生态库名前缀' | head -10
```

**生态链审查要点：**
- 修改 L0/L1 的导出符号 → 必须确认 L2/L3 调用者兼容
- 新增功能放在了错误的层级 → 标记为架构问题
- 后端 API 变更 → 前端项目是否需要同步

- **ctx 契约**（后端）：context 是否在边界处正确透传？

## Step 3: 逐项深度检查

### 3.1 业务逻辑
- 功能实现是否正确，条件判断 / 状态流转 / 数据处理是否有错误
- 是否引入回归问题
- **数值验算**：超时预算、退避公式、限流阈值等，用生产配置值代入计算

### 3.2 前后端契约对齐（重点）
- 接口字段名、类型、枚举值、默认值是否前后端一致
- 状态码 / 错误码的含义和处理是否对齐
- 校验规则是否前后端同步（前端校验 ≠ 后端不校验）
- 是否存在：前端已调用但后端未实现 / 后端已返回但前端未消费 / 字段理解不一致
- **新增/修改的 API 参数，前端是否已适配？常量定义是否同步？**

### 3.3 功能闭环（重点）
- 业务流程是否能从入口走到最终结果（不存在流程断点）
- 成功路径和失败路径是否都完整处理
- 是否存在"代码存在但业务未闭环"的情况：
  - 页面可操作但流程走不通
  - 数据未正确落库
  - 状态未回写
  - 操作后无反馈
  - 前后端只有一侧实现

### 3.3.1 架构设计审查（高层）
- **SOLID / 分层检查**：
  - 是否单一职责，还是一个模块同时做路由、业务、缓存、序列化、外部调用
  - 新功能扩展是否主要靠修改旧 `if/else`、`switch`，还是能通过扩展点完成
  - 高层模块是否依赖抽象而不是具体实现，依赖是否可被测试替换
- **耦合 / 内聚**：
  - 是否把完整大对象到处传递，但实际只用其中少数几个字段
  - 是否通过布尔开关参数控制大量行为分叉，导致控制耦合
  - 同一模块的方法是否都围绕单一业务目标，还是“什么都干一点”
- **反模式 / 过度设计**：
  - 是否出现 God Object、Big Ball of Mud、复制粘贴扩散、只有一个实现却强行抽接口
  - 是否为了“以后可能要扩展”引入复杂策略模式、工厂、注册表，但当前没有真实需求
  - 对 reviewer 来说，要明确区分：必要抽象、过度抽象、兼容性妥协
- **YAGNI 检查**：
  - 新增 endpoint / DTO / store / hook / 配置项是否真的被调用
  - 如果外部 review 建议“做完整一点”，先 grep/追调用，确认不是无人使用的船锚代码

### 3.4 后端 Go 专项（递归到叶子）

- **goroutine 生命周期**：泄漏 / channel 未关闭 / select 死锁
  - goroutine+select+time.After 是已知反模式：外层放弃等待 ≠ 底层停止
- **context 传播（三向递归审查 + 完成度验证）**：
  - **向外**：所有调用者是否传递了正确的 ctx？包括其他分支上的调用者
  - **向内（递归到叶子，覆盖所有分支）**：函数内部的每个耗时操作是否继承 ctx？**降级/fallback 分支中调用的 helper 也要追**
  - **跨层**：ctx 能否从最上游传到最下游？常见断层：mock context / goroutine 内 Background() / 回调无 ctx / 接口无 ctx
  - **完成度验证**：`grep -c 'context.Background()' <文件>` 逐条确认
- **error 处理**：吞错误 / 未 `%w` 包装 / error 链断裂
- **sync 原语**：Once/Mutex/WaitGroup 使用正确性，sync.Once 在测试中的 -count=N 陷阱
- **内存**：io.ReadAll 无 LimitReader / 大 slice 未释放 / 闭包泄漏

### 3.5 前端专项

- **状态管理**：是否正确更新状态 / 是否有过期状态 / 竞态
- **错误处理**：API 错误是否 catch / 是否显示给用户 / 是否区分网络错误和业务错误
- **内存泄漏**：useEffect 清理 / 定时器清理 / 事件监听移除
- **渲染性能**：不必要的 re-render / 大列表无虚拟化 / 重复 API 调用
- **Promise/异步链**：
  - 是否存在 `.then().catch()` 吞错、重复请求、竞态覆盖旧响应
  - abort/cancel 是否真正传到底层请求，而不是只更新上层 loading 状态
- **状态副作用审计**（参考 click-path-audit）：
  对变更涉及的 state store（Zustand/Redux/Context/useState），映射每个 action 的显式写入和**隐式重置**，然后对每个交互入口（onClick/onSubmit），按顺序追踪 handler 中的函数调用，检查 6 类 bug 模式：
  1. **Sequential Undo**：函数 B 的副作用撤销了函数 A 刚写入的状态
  2. **Async Race**：两个异步调用的最终状态取决于 resolve 顺序
  3. **Stale Closure**：useCallback/useMemo 捕获了过时的变量
  4. **Missing State Transition**：按钮有 handler 但实际没有执行核心操作（只校验不保存、只设 flag 不调 API）
  5. **Conditional Dead Path**：条件永远为 false，核心逻辑永远不执行
  6. **useEffect Interference**：useEffect 监听了刚被设置的状态并立即重置
- **性能阈值意识**：
  - 列表超过约 100 项是否需要分页或虚拟滚动
  - 大型模块/图表/PDF/管理后台是否应该懒加载或按路由分包
  - 图片/媒体资源是否使用合适格式和尺寸，避免首屏关键资源被 lazy load

### 3.5.1 前端性能 & Core Web Vitals
- 是否引入对首屏有明显影响的资源：
  - LCP 元素是否被懒加载、低优先级加载或被大体积资源拖慢
  - 是否有可能导致布局抖动的图片、广告位、异步插入内容

### 3.7 安全
- 后端：SQL 注入、命令注入、越权、鉴权缺失、敏感信息泄露
- 前端：XSS、CSRF、敏感信息在前端暴露、localStorage 存敏感数据
- 认证/授权专项：
  - 每个新增接口都要确认鉴权和资源归属校验，不只看“是否登录”
  - 是否存在 IDOR、越权读写、测试环境绕过逻辑误带到正式逻辑
- 输入/输出专项：
  - 富文本、HTML 片段、文件名、路径、命令参数、SQL 条件是否被安全处理
  - 错误响应、日志、前端提示是否暴露栈、SQL、密钥、内部 URL
- 配置/部署专项：
  - Secret 是否落在 Secret 而非 ConfigMap
  - debug、profiling、内部端口、测试开关是否误带到生产
  - 外部可访问端点是否有速率限制、CORS、Header 基线

### 3.7.1 API / 数据安全闭环
- 列表接口是否强制分页、限制每页最大数量、避免 `SELECT *` 或返回过宽字段
- 热点/公开接口是否有速率限制与缓存策略
- 敏感字段是否在后端脱敏，前端是否只消费必要字段
- 上传/下载/导出能力是否校验身份、租户、对象归属和过期时间

### 3.8 并发 & 稳定性（后端）
- 竞态条件（`go test -race`）
- 连接池耗尽、goroutine 爆炸、OOM 风险
- 资源释放完整性
- **超时后资源归属**：超时返回后底层 goroutine / 连接还在跑吗？
- **数据库 / API 性能**：
  - 是否存在 N+1 查询、缺分页的大列表、热接口无缓存、无索引过滤、大量 `SELECT *`
  - 新增循环、嵌套查询、`includes/find` 这类 O(n²) 模式是否在真实数据量下可接受
- **前后端联动性能**：
  - 前端是否导致重复轮询、无节制重试、并发请求风暴
  - 后端是否在同一请求中串行执行本可并行的外部调用

### 3.9 多 Pod / K8s 安全（后端）
- 是否依赖单 Pod 本地状态
- 外部副作用是否幂等
- SIGTERM 优雅停机

### 3.10 变更辐射区扫描（防遗漏）

- **同文件邻居函数**：修了 A 的问题，B/C 是否有相同问题？
  ```bash
  grep -n 'context.Background()' <被修改的文件>  # 同文件残留
  ```
- **同包同类函数**：同包功能类似的函数
- **接口的所有实现**：`grep -rn 'func.*) MethodName(' --include='*.go'`
- **其他分支的代码**：dev/main 上独有的调用者，合并后是否兼容
- **前端常量同步**：后端新增的枚举值 / 渠道类型，前端 `constants/` 是否同步
