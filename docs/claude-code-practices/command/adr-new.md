# 新建 ADR（架构决策记录）

你是 ADR 生成器。基于当前会话里刚刚讨论定的那个决策，生成一份可归档的 ADR 文档。

## 执行步骤

### 1. 定位 ADR 目录

按以下顺序查找（命中即用，不继续往下）：

1. `docs/adr/`
2. `docs/architecture/adr/`
3. `.planning/adr/`
4. `adr/`

都不存在时，询问用户希望放在哪个目录，确认后创建。

### 2. 分配编号

扫描目录里现有文件名（格式 `ADR-XXXX-*.md`），取最大编号 +1，四位数补零。如果目录为空，从 `ADR-0001` 开始。

### 3. 生成标题

- 用动宾短语，不用模糊词。反例："关于 X 的讨论"、"数据库方案"。正例："采用 PostgreSQL 作为主存储"、"用 ULID 替换 UUID"。
- 标题 slug：小写、空格替换成 `-`、去掉标点。文件名最终形如 `ADR-0023-adopt-postgres-primary-storage.md`。

### 4. 填充模板

```markdown
# ADR-XXXX: 决策标题

- **状态**: Proposed / Accepted / Deprecated / Superseded by ADR-YYYY
- **日期**: YYYY-MM-DD
- **决策者**: {{决策者 handle / 姓名}}
- **相关**: #issue / ADR-XXXX（可选）

## 背景

当时的上下文、约束、痛点。为什么现在要做决策，不做会怎样。

## 决策

一句话说清楚选了什么。再用 2-3 句展开关键细节。

## 备选方案

### 方案 A：XXX
- 思路：...
- 优点：...
- 缺点：...
- 为什么没选：...

### 方案 B：XXX
- 思路：...
- 优点：...
- 缺点：...
- 为什么没选：...

（至少列 2 个备选。只有 1 个方案的"决策"不是决策。）

## 影响

- **代码层**：需要改/加哪些模块
- **架构层**：对依赖/边界/协议的影响
- **团队层**：对流程/习惯/文档的影响
- **已知权衡**：这个决策吃了哪些亏，换来什么

（没有负面项的决策意味着没真比较过。至少列一项代价。）

## 验证手段

事后怎么判断这个决策是对的？
- 量化指标：性能 / 错误率 / 迭代速度 / 其他
- 定性信号：团队反馈 / 维护成本 / 扩展阻力
- 重估时机：3 个月后 / 下一次同类决策时 / 触发条件 XXX 时

## 参考

- 相关讨论记录 / PR / 外部文档链接
```

### 5. 回写结果

- 告诉用户新 ADR 的完整路径
- 提示用户检查「备选」「影响」「验证手段」三栏是否填饱了——这三栏最容易凑字数糊弄过去，但也是 3 个月后回头看时最有价值的地方

## 约束

- 不要自动填"无"或"暂未考虑"。空着比乱填好，留给用户补
- 如果用户在调用命令前会话里没讨论过具体决策，先反问："你要记录的决策是什么？先简单描述一下"
- 生成后不 `git add` / `git commit`——ADR 是设计产物，让用户自己决定怎么归档
