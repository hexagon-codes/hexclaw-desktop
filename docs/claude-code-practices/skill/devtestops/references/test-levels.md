# 变更类型→测试级别映射与提测准入基线

## 变更类型判定规则

根据 `git diff` 和变更文件范围判定变更类型：

| 变更类型 | 判定标准 |
|---------|---------|
| 纯逻辑调整 | 仅修改单文件/单函数内部逻辑，不涉及接口签名、跨模块调用或数据模型变更 |
| 接口改动 | 新增或修改 API 端点、请求/响应字段、错误码、鉴权方式 |
| 跨模块改动 | 变更涉及 2 个以上模块/包，或修改了共享数据模型、公共工具函数 |
| 核心链路改动 | 变更涉及登录/鉴权、下单/支付、任务提交/回调等核心业务入口 |
| 高风险改动 | 涉及数据迁移、架构重构、并发模型变更、第三方服务切换 |

## 测试级别定义

### 单元测试（P0）

覆盖最小可测单元的正常/异常/边界路径。

**测试框架**：根据项目技术栈自动识别（Vitest / Jest / pytest / Go test 等）

**覆盖要求**：
- 函数/方法的正常输入 → 正确输出
- 边界输入：空值、零值、超长、非法格式、特殊字符
- 异常路径：错误输入、非预期状态、字段缺失
- 状态流转：合法状态切换 + 非法状态处理

**Claude Code 执行方式**：
- 使用 Bash 工具执行 `npx vitest run` / `go test ./...` / `pytest`
- 使用 Grep 工具扫描未覆盖的核心函数

### 模块测试（P0）

组件/模块与直接依赖的联动行为。

**覆盖要求**：
- 组件 + Store/State 联动
- 组件 + 路由守卫联动
- Service + Repository 联动
- 中间件/拦截器行为验证

### 接口/API 测试（P1）

**覆盖要求**：
- 入参校验（必填、类型、格式、长度）
- 出参结构（字段完整性、类型正确性）
- 状态码和错误码
- 鉴权/权限（未登录、越权、角色差异）
- 幂等性（重复请求）
- 分页和过滤逻辑
- 兼容性（老字段不删、新字段有默认值）

**Mock 契约验证**（Demo/前端项目）：
- Mock 数据字段与 PRD 数据模型一致
- Mock 文件有 @prd 标注
- 调用处有 TODO[接入] 标注
- TypeScript 类型定义与 PRD 对齐

### 集成测试（P1）

**覆盖要求**：
- 跨模块调用链完整跑通
- 数据在模块间正确传递
- 事务一致性（DB 事务成功/失败/回滚）
- 缓存与 DB 一致性
- 消息队列生产者→消费者

### E2E 测试（P2）

**覆盖要求**：
- 从真实入口走完整业务流程
- 浏览器/客户端级别操作
- 跨页面/跨服务的状态流转
- 响应式布局验证

**Claude Code 执行方式**：
- 使用 Playwright MCP 进行浏览器自动化
- 使用 Browser MCP 进行截图比对

### 系统测试（P2）

**覆盖要求**：
- 跨模块业务流程闭环（如：注册→审核→登录→使用）
- 多角色交叉操作
- 数据隔离验证

### 安全测试（P1）

**覆盖要求**：
- XSS 防护（输入转义、输出编码）
- 注入防护（SQL 注入、命令注入）
- 权限绕过（URL 篡改、角色提权）
- 会话管理（持久化、过期、多标签同步）
- 敏感信息泄露（密码明文、日志、URL、响应体）

### 兼容性与回归测试（P2）

**覆盖要求**：
- 历史数据兼容（老数据不报错）
- 浏览器兼容（Chrome / Firefox / Safari）
- 移动端兼容
- 修复后回归不引入新问题

### 性能/基准测试（P2）

**覆盖要求**：
- 关键路径响应时间基线
- 并发压力下的稳定性
- 内存/CPU 使用无异常增长

## 提测准入基线

| 变更类型 | 必须完成 | 建议补充 | 不满足时 |
|---------|---------|---------|---------|
| 纯逻辑调整 | 单元 + 模块 + 基础回归 | 边界值 + 异常路径 | 🔴 不建议提测 |
| 接口改动 | + API 测试 + 兼容性检查 | 错误码 + 鉴权 | 🔴 不建议提测 |
| 跨模块改动 | + 集成测试 + 数据一致性 | 超时重试 | 🔴 不建议提测 |
| 核心链路改动 | + E2E + 专项测试 | 上线验证清单演练 | 🔴 不建议提测 |
| 高风险改动 | + 性能基准 + 回滚演练 + 灰度方案 | 异常场景演练 | 🔴 原则上不提测 |
